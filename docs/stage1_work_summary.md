# Stage 1 Work Summary

## Цель этапа

Собрать первый рабочий этап проекта по анализу платежей из SalesDrive API так, чтобы:
- загружать входящие и исходящие платежи за период
- поддерживать пагинацию
- хранить и анализировать raw API response
- нормализовать платежи во внутреннюю модель
- отделять внутренние перемещения от реальных поступлений
- классифицировать входящие поступления
- группировать исходящие платежи по контрагентам и поставщикам
- формировать понятный Excel-отчет

### 4. Анализ реального API

Выполнен живой анализ ответа SalesDrive API.

Подтверждено, что top-level response содержит:
- `data`
- `pagination`
- `totals`
- `status`

Ключевые поля платежа:
- `id`
- `date`
- `sum`
- `purpose`
- `type`
- `counterparty.title`
- `counterparty.egrpou`
- `organization.title`
- `organization.egrpou`
- `organizationAccount.accountNumber`

Зафиксированные ограничения API:
- валюта в item payload не приходит
- item-level статус платежа не приходит
- `comment` часто пустой, основной смысл платежа находится в `purpose`
- у части платежей контрагент не заполнен отдельным объектом и виден только в тексте `purpose`

Результат зафиксирован в:
- `docs/payment_api_analysis.md`
- raw JSON в `debug/`

## Логика бизнес-обработки

### 6. Внутренние перемещения

Реализован отдельный механизм определения внутренних перемещений.

Источник правил:
- `config/internal_transfer_rules.yaml`

В правила зафиксированы:
- собственные счета
- собственные сущности
- допустимые пары счетов для self-transfer
- текстовые маркеры внутренних переводов

Используемая логика:
- если есть входящий и исходящий платеж
- суммы совпадают
- разница во времени укладывается в окно
- счета входят в список своих счетов
- в тексте есть self-transfer маркеры
- тогда такие платежи фиксируются как внутренняя пара

Также реализован fallback на direct self markers без зеркальной пары.

По реальным данным:
- за март 2026 найдено 16 пар внутренних переводов
- дополнительно найден 1 прямой self inflow без зеркальной пары

### 7. Классификация входящих поступлений

Введены категории:
- `customer_receipt`
- `excluded_receipt`
- `internal_transfer`

Логика:
- внутренние перемещения исключаются первыми
- затем применяются правила customer receipts из `config/analysis_settings.yaml`
- отдельные исключения используются для возвратов и нецелевых поступлений

На основе живых данных были выделены:
- агрегированные клиентские поступления через `НоваПей`
- прямые клиентские входящие платежи по текстам назначения
- возвраты как отдельный класс исключений

### 8. Мапинг контрагентов к поставщикам

Реализован каскадный механизм:
1. Excel mapping
2. YAML mapping
3. `unmapped`

Важно:
- Excel считается источником истины для `supplier_name`
- именно значения из Excel используются для:
  - supplier summary
  - названий supplier sheets
  - supplier_name внутри отчета

## Отчетность

### 9. Excel-отчеты

Собирается Excel workbook с листами:
- титульная сводка
- входящие платежи
- поступления от клиентов
- исключенные поступления
- внутренние входящие
- исходящие платежи
- внешние исходящие
- внутренние исходящие
- пары внутренних переводов
- агрегаты по контрагентам
- агрегаты по поставщикам
- unmapped
- errors
- отдельные листы по каждому поставщику

### 10. Что есть на титульном листе

На `summary_Сводка` выведено:
- основные KPI по месяцу
- количество и сумма
- отдельный блок по поставщикам:
  - поставщик
  - количество исходящих платежей
  - сумма исходящих платежей

### 11. Supplier sheets

Для найденных поставщиков создаются отдельные листы.

В каждом листе:
- итог по поставщику
- разбивка по дням
- детальная таблица платежей

Названия листов берутся из `supplier_name` в Excel mapping.
